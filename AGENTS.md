# Agent Instructions

## Running Python Commands

Always use `uv run` to run Python scripts and commands in this project. For example:

```bash
uv run python tools/merge_lora.py ...
uv run python -m fish_speech.train ...
```

## Experiment Naming Convention

All experiments are named with the speaker and a datetime in ISO format to avoid collisions:

```
<speaker>_<YYYY-MM-DDTHHMMSS>
```

Example: `barbero_2026-03-20T083000`

This name is used consistently as:
- The training `project` name → `results/barbero_2026-03-20T083000/`
- The merged model directory → `checkpoints/s2-pro-barbero_2026-03-20T083000-<step>/`
- Audio samples (generated by callback during training) → `results/barbero_2026-03-20T083000/audio_samples/step_<N>.mp3`

Always get the current datetime before starting an experiment and use it throughout.

## Training Workflow

### Launching a run

```bash
EXP=barbero_$(date +%Y-%m-%dT%H%M%S)
mkdir -p results/${EXP}
PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True uv run python fish_speech/train.py \
    --config-name text2semantic_finetune \
    project=${EXP} \
    +lora@model.model.lora_config=<lora-config> >> results/${EXP}/train.log 2>&1 &
```

The `mkdir -p` ensures the log redirect target exists before the process starts. The `2>&1` captures the tqdm progress bar (stderr) into `train.log` alongside the Python logger output.

### Saving configs to the experiment folder

After training starts (hydra creates `results/<experiment>/` immediately), copy the source configs there for reproducibility. Use the exact experiment name from the command above:

```bash
EXP=barbero_<YYYY-MM-DDTHHMMSS>
cp fish_speech/configs/text2semantic_finetune.yaml results/${EXP}/
cp fish_speech/configs/lora/<lora-config>.yaml results/${EXP}/
```

Note: hydra also saves a fully resolved config at `results/${EXP}/.hydra/config.yaml`, but keeping the source files makes it easy to reuse the exact setup for a future run.

## LoRA Merge Workflow

Audio samples are generated automatically during training by `AudioSampleCallback` and saved to `results/<experiment>/audio_samples/`. Use those to identify the best checkpoint, then merge it for final evaluation or publishing.

### Merge LoRA checkpoint into base model

Output path must include the base model name, the experiment name, and the step.

**The lora config used for merging must exactly match the one used during training** — including `target_modules`. The safest source of truth is the config file saved to the experiment folder (`results/<experiment>/<lora-config>.yaml`). If that file has no `target_modules`, the checkpoint contains all modules (embeddings, output, attention, mlp). Using a config with a different `target_modules` will cause a `RuntimeError` on mismatched keys.

If the saved config is not in the lora configs directory, copy it there or create a matching one before merging.

```bash
uv run python tools/llama/merge_lora.py \
    --lora-config <lora-config> \
    --base-weight checkpoints/s2-pro \
    --lora-weight results/<experiment>/checkpoints/<step>.ckpt \
    --output checkpoints/s2-pro-<experiment>-<step>/
```

Example: `--lora-weight results/barbero_2026-03-20T083000/checkpoints/step_000001000.ckpt` → `--output checkpoints/s2-pro-barbero_2026-03-20T083000-step_000001000/`

**Important:** the merge script OOMs during its validation step (torch.load of the saved model) but the merged model is saved successfully before that. If the script exits with code 137, check that the output directory exists and proceed to inference — do not retry the merge.

## Dataset Configuration

### Causal vs. random sampling (`causal` flag)

In `fish_speech/configs/text2semantic_finetune.yaml`, both `train_dataset` and `val_dataset` have a `causal` flag:

- `causal: true` — picks a random start index and returns N **consecutive** clips from the group. If the training data comes from long sessions (e.g. 1-hour recordings), clips from the same session will be fed together for many steps before moving to the next session, which can cause uneven training.
- `causal: false` — randomly samples N clips from across the entire group, regardless of session order. Use this when data comes from multiple long sessions to ensure proper mixing.

For the barbero dataset (clips from multiple 1-hour sessions, all in a single proto group), use `causal: false`.
